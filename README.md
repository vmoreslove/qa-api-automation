# Selenium UI Tests for `qa-test.html`

Небольшой учебный Maven-проект с UI-автотестами на Java/Selenium для локальной HTML-страницы [`qa-test.html`](/Users/alex/work/oleg/studyrepo/qa-test.html). Проект демонстрирует базовую организацию UI-проверок через Page Object и JUnit 5.

## Что тестируется

Тестовый стенд содержит два сценария:

1. Авторизация по `email` и `password`.
2. Заполнение формы пользователя с последующим добавлением записи в таблицу.

Логика страницы:

- корректный логин: `test@protei.ru`
- корректный пароль: `test`
- при успешной авторизации скрывается блок `authPage` и показывается `inputsPage`
- при отправке формы выполняется валидация email и имени
- после успешной отправки запись добавляется в таблицу `dataTable`
- пользователь видит модальное сообщение `Данные добавлены.`

## Стек

- Java 11
- Maven
- Selenium Java `4.15.0`
- WebDriverManager `5.6.2`
- JUnit 5
- Google Chrome / ChromeDriver

- [`src/test/java/tests/DebugTest.java`](/Users/alex/work/oleg/studyrepo/src/test/java/tests/DebugTest.java) — вспомогательный тест для пошаговой отладки авторизации и снятия скриншотов.

## Покрытые сценарии

### `FormTests`

Позитивные сценарии создания пользователя:

- создание пользователя с выбранными чекбоксами `1.1` и `1.2`
- создание пользователя с радиокнопкой `2.1`
- создание пользователя с радиокнопкой `2.3`

Во всех тестах проверяется, что после отправки формы email пользователя появился в таблице.

### `DebugTest`

Отладочный сценарий авторизации:

- ищет элементы формы логина
- вводит учетные данные
- нажимает кнопку входа через JavaScript
- делает скриншоты до и после нажатия
- проверяет, что форма логина исчезла и появилась основная форма

## Как запустить

### Требования

- установлен JDK 11+;
- установлен Maven;
- установлен Google Chrome;
- есть доступ в интернет для загрузки ресурсов, подключенных в `qa-test.html` через CDN;
- путь к локальному файлу `qa-test.html` совпадает с тем, что указан в тестах, либо предварительно исправлен.

### Команда запуска

```bash
mvn test
```

## Важное ограничение проекта

Сейчас тесты привязаны к абсолютным путям, которые захардкожены в коде:

- в [`src/test/java/tests/FormTests.java`](/Users/alex/work/oleg/studyrepo/src/test/java/tests/FormTests.java) для открытия страницы используется `file:///Users/olegkopylov/IdeaProjects/Protei/qa-test.html`
- в [`src/test/java/tests/DebugTest.java`](/Users/alex/work/oleg/studyrepo/src/test/java/tests/DebugTest.java) используется тот же путь к HTML-файлу и абсолютные пути для сохранения скриншотов

Из-за этого проект не переносим "как есть" между машинами. В текущем состоянии запуск `mvn test` в другом окружении приводит к ошибкам поиска элементов страницы, потому что Selenium открывает несуществующий или нерелевантный файл.

## Что показал анализ проекта

Сильные стороны:

- простая и понятная структура;
- используется Page Object;
- тестовые сценарии легко читать и расширять;
- есть отдельный отладочный тест для проблемной авторизации.

Текущие технические ограничения:

- абсолютные пути в тестах;
- зависимость от внешних CDN-ресурсов в локальной HTML-странице;
- использование `Thread.sleep(...)` вместо полностью управляемых ожиданий;
- покрыты только позитивные сценарии;
- авторизация реализована через JavaScript-клик как обходной путь, а не через более устойчивое ожидание кликабельности элемента.

## Что можно улучшить

- вынести путь к `qa-test.html` в параметр, конфиг или системное свойство;
- сохранять скриншоты в каталог проекта, а не в пользовательский абсолютный путь;
- добавить негативные тесты:
  - неверный email при логине;
  - неверный пароль;
  - пустое имя;
  - невалидный email в основной форме;
- убрать лишние `Thread.sleep(...)` и заменить их на явные ожидания;
- добавить параметризацию тестовых данных и путей запуска.

## Итог

Проект подходит как учебный пример UI-автотестов на Selenium для локальной HTML-формы. Основная архитектура уже есть, но для нормального переиспользования и запуска на любой машине нужно убрать абсолютные пути и немного стабилизировать работу тестов.
